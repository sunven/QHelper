## Problem Statement

Migrate `/Users/sunven/github/text-preview` into QHelper as a first-class Chrome extension tool. The migrated tool should preserve the original value: paste or open local text, extract copyable IPs, URLs, Unix paths, and shell commands locally, and keep multi-tab workspace state in the current browser.

The target is a developer utility inside QHelper, not a separate product surface. The user selected:

- Full feature migration.
- Developer-tool quality bar.
- Tool-internal full-screen workspace layout.
- QHelper-native architecture.

## What Makes This Cool

This turns QHelper from a set of formatters into a practical ops/debugging workbench: paste logs, configs, commands, notes, or local files, then copy exactly the useful objects without sending content anywhere. The "whoa" moment is opening a messy text dump and immediately getting a clean, copyable list of IPs, commands, paths, and URLs, grouped by source line.

## Constraints

- Keep Chrome/Chromium extension behavior compatible with WXT MV3.
- Use Node 22 and pnpm in QHelper.
- Do not edit generated `.output/` or `.wxt/`.
- Keep domain logic outside React components.
- Preserve the privacy model: no backend, no AI/API calls, no third-party analytics, no remote scripts for pasted or opened content.
- File System Access must remain user-triggered and capability-detected; browsers without support still allow paste and copy.
- Text Preview opts out of QHelper's keep-alive route behavior; workspace persistence is the recovery mechanism when navigating away and back.
- Human-readable gstack artifacts for this repo live under `docs/gstack/`.

## Premises

1. Register Text Preview as a normal QHelper tool so it appears in popup/sidebar/catalog and resolves at `tools/<id>.html`.
2. Preserve the original local-only model: parser functions are pure TypeScript, workspace content stays in current-browser storage, and "clear all" removes saved workspace plus file handles.
3. Preserve the original user workflows: multi-tab paste buffers, sample fill, parser warnings, result copy, copy-all, manual fallback on clipboard failure, and debug mode via query param.
4. Preserve local file workflows in Chrome/Edge: open one or more small text files, save current file, save all, detect external disk changes before overwrite, reload, force overwrite, and download fallback copy.
5. Allow new dependencies required for the original experience: CodeMirror 6 and `react-resizable-panels`; add only the UI primitives needed by this tool.

## Approaches Considered

### Approach A: QHelper-native full migration

Move parser, workspace, clipboard, local file access, and file handle storage into `lib/text-preview/`. Move UI into `components/tool/text-preview/` and register a `text-preview` tool component. Keep it reachable through the normal tools SPA, but make the tool content itself a dense full-height workbench.

Effort: L  
Risk: Medium

Pros:

- Matches QHelper's existing split between `lib/` domain logic and React UI.
- Preserves the full original capability set and makes tests straightforward.
- Keeps popup, sidebar, route aliases, and catalog behavior consistent with other tools.

Cons:

- Largest initial diff.
- Requires dependency and lockfile updates.
- Needs careful height and scroll handling inside `ToolWorkspaceShell`.

### Approach B: Fast component port

Copy the original app tree nearly as-is into one tool component area, then patch imports, styles, and dependencies until it runs inside QHelper.

Effort: M  
Risk: Medium

Pros:

- Fastest path to a visible working tool.
- Lowest risk of accidentally changing parser and workspace behavior.

Cons:

- Leaves a mini-app structure inside QHelper.
- Makes future parser and file-workflow maintenance harder.
- Tests will not line up cleanly with QHelper's current conventions.

### Approach C: Standalone extension page

Add a separate WXT extension page that closely mirrors the original Vite app and link to it from QHelper.

Effort: M  
Risk: High

Pros:

- Preserves original full-screen feel most directly.
- Keeps the original app boundary easiest to split back out later.

Cons:

- Conflicts with the selected premise that this should be a normal QHelper tool.
- Creates a separate route and testing path.
- Weakens sidebar/catalog state consistency.

## Recommended Approach

Use Approach A.

The implementation should feel like QHelper owns this tool, while the inner workbench remains optimized for long text. Concretely: the route, catalog, popup, sidebar, and alias generation stay standard; the Text Preview component uses a full-height, split-pane layout inside the tool content area.

## Implementation Plan

1. Add dependencies and UI primitives.
   - Add CodeMirror 6 packages used by the original `SourceTextEditor`.
   - Add `react-resizable-panels`.
   - Add QHelper-style `badge`, `alert`, `tabs`, `empty`, `resizable`, and `scroll-area` primitives only if they do not already exist.
   - Keep styling aligned with current QHelper tokens, square corners, dense controls, and JetBrains Mono.

2. Move domain logic into `lib/text-preview/`.
   - Port parser modules: `pipeline`, `types`, `id`, `line`, `sizeGuard`, `limits`, `conflictResolver`, and parsers for IP, URL, Unix path, and shell command.
   - Port workspace modules: workspace state transitions, storage persistence, local file access, file handle IndexedDB store, file system access types, sample text, and clipboard helper.
   - Keep parser modules pure: no DOM, storage, clipboard, network, or React imports.
   - Use a storage key namespaced for the extension, for example `qhelper.text-preview.workspace.v1`; do not reuse generic keys from unrelated tools.
   - Use direct tool-page `window.localStorage` for workspace persistence. Do not use `lib/chrome/storage.ts`, because that wrapper writes `chrome.storage.local` first and expands quota/privacy exposure for large pasted text.
   - Use a QHelper-namespaced IndexedDB database for file handles, for example `qhelper-text-preview-file-handles`; do not reuse the standalone app's `text-preview-file-handles` database name.

3. Build the QHelper tool UI.
   - Add `components/tool/text-preview/` with the original workbench split into focused components: toolbar, tabs, source editor, result list, status messages, file status panel, fallback value, and debug panel.
   - Wrap the top-level export as `App` so it matches existing tool route imports.
   - Use `ToolPageShell` with `toolId="text-preview"` for identity, but make the content fill available height.
   - Keep the original responsive behavior: side-by-side panes on desktop, input above results on narrow screens.
   - Keep trust signals visible near the top: local parsing, no upload, current-browser save, clear-all cleanup.

4. Register the tool everywhere QHelper expects it.
   - Add metadata to `lib/registry/tools.ts` with id `text-preview`, Chinese name `文本预览`, English name `Text Preview`, category `COMMON`, icon `FileText` or equivalent, and entry `tools/text-preview.html`.
   - Add metadata support for `preserveActivity?: boolean`, defaulting to `true`.
   - Set `preserveActivity: false` for `text-preview` so large CodeMirror documents are not kept mounted while hidden.
   - Add the component to `components/tool/tool-routes.tsx`.
   - Update `ToolActivityOutlet` so hidden tools with `preserveActivity === false` are unmounted instead of wrapped in hidden React `Activity`.
   - Let existing catalog, sidebar, popup, and WXT alias generation derive the new route from metadata.

5. Preserve local file behavior.
   - `打开文件` uses `showOpenFilePicker` only after the user clicks.
   - Reject files over 1MB before reading or saving to workspace.
   - Store file handles in IndexedDB separately from workspace state.
   - Before saving, re-read the current disk file and compare content hash with the last known signature.
   - Conflict state offers reload, force overwrite, and download copy.
   - Save-all reports saved, skipped-clean, conflict, permission-needed, handle-missing, and failed items.

6. Preserve privacy and failure behavior.
   - No network calls from Text Preview code.
   - Clipboard failures reveal a read-only textarea containing the exact value to copy.
   - Storage failures show local-save warnings without blocking parsing.
   - Clearing workspace removes the workspace storage key and clears the file handle store.
   - Debug panel is hidden unless `?debug` is present.

## Success Criteria

- `chrome-extension://<id>/tools/text-preview.html` loads from the tools SPA.
- Popup and sidebar show the tool and open the generated alias route.
- Filling the sample displays at least `192.168.80.45` and `ls -l $(which kubectl)` as copyable results.
- Multi-tab workspaces isolate text, result counts, rename state, close confirmation, and restore after refresh.
- Navigating from Text Preview to another QHelper tool unmounts the Text Preview component, then restores its workspace from `qhelper.text-preview.workspace.v1` when revisited.
- Local file-backed tabs can be opened, edited, saved, reloaded, and conflict-checked in Chromium.
- Browsers without File System Access show the unsupported warning and still support paste, parse, and copy.
- Pasted/file-loaded sentinel text is not observed in network request URL, headers, or body during E2E.
- Pasted/file-loaded sentinel text is stored only in `window.localStorage` under `qhelper.text-preview.workspace.v1`; it is not written to `chrome.storage.local`, `sessionStorage`, or unrelated keys.
- `清空全部` removes saved workspace text and file handle storage.

### Pass 1: Information Architecture

Score: 6/10 -> 9/10 after fixes.

Text Preview must use this hierarchy:

```text
QHelper tools shell
  |-- persistent sidebar navigation
  |-- breadcrumb/header: QHelper Tools / category / 文本预览
  `-- Text Preview workbench
      |-- workbench toolbar
      |   |-- left: title "文本预览" + compact trust chips
      |   |-- right: 打开文件 / 保存 / 保存全部 / 填入示例
      |   `-- optional info popover: local parsing, no upload, browser-only save, clear-all cleanup
      |-- workspace tab strip
      |   |-- horizontally scrollable manual/file-backed tabs
      |   |-- object count and file status badges per tab
      |   |-- new tab action near tabs
      |   `-- 清空全部 at the far edge, visually destructive
      `-- split workspace
          |-- input pane
          |   |-- file status or conflict banner, only when relevant
          |   |-- CodeMirror editor, line numbers, wrapping
          |   `-- parser/storage warnings close to editor
          `-- result pane
              |-- sticky summary row: total objects + copy-all
              |-- grouped results in fixed order
              |-- manual copy fallback inline with failed copy
              `-- debug panel only when ?debug is present
```

First-screen priority:

1. Active editing context: selected tab, source text area, and current result count.
2. Primary actions: paste/type, open file, copy results, save file-backed edits.
3. Trust and safety: local parsing, no upload, current-browser save, clear-all cleanup.

The trust signal must be visible but compact. Use chips such as `本地解析`, `不上传`, `本机保存`; place detailed privacy text behind an info button or disclosure so it does not compete with the editor.

Result pane information architecture:

- Fixed group order: `IP 地址`, `URL`, `命令`, `路径`.
- Hide empty groups unless every group is empty.
- Each group header shows count and a scoped copy action when useful.
- Rows are line-grouped first, then object-grouped within the line, so users can map results back to source text quickly.
- Copy buttons stay adjacent to the value they copy; no shared distant action for individual rows.

Mobile/narrow information architecture:

- Keep the QHelper shell header.
- Toolbar wraps into two rows maximum: title/trust first, actions second.
- Tab strip remains horizontally scrollable.
- Stack input above results; copy controls remain visible without horizontal scrolling.

| Workspace | First load / empty workspace | One `未命名 1` tab, empty editor, result pane empty state | Paste text, click `填入示例`, or `打开文件` |
| Editor | Empty input | Placeholder in editor plus result-pane empty state explaining supported objects | Paste/type or fill sample |
| Parser | No extracted results | Result pane says `没有识别到可复制对象。` and names supported object types | Edit input or clear tab |
| Parser | Large input warning | Compact warning near editor, not a blocking modal | Continue editing; parser still runs |
| Parser | Hard stop >1MB | Warning near editor, result pane shows no objects | Reduce input or split into tabs/files |
| Parser | Partial parser failure | Warning near editor naming failed parser modules | Keep available results visible |
| Results | Results available | Sticky result summary with total count and copy-all; line-grouped rows below | Copy row value or copy all |
| Clipboard | Copy success | Button label/state changes to `已复制` for about 1.5 seconds and polite live region announces it | Continue working |
| Clipboard | Copy failure | Inline alert/fallback textarea containing exact copy value | User manually selects/copies value |
| Workspace storage | Storage unavailable/corrupt | Compact local-save warning near editor | Continue parsing; warn that refresh may lose content |
| Workspace tabs | Close non-empty tab first click | Delete button switches to confirm state; live region says click again to delete | Click again to delete or wait/cancel by doing nothing |
| Workspace tabs | Max tabs reached | New tab button disabled with tooltip/title explaining the limit | Close an existing tab |
| File access | Unsupported browser | Compact warning near editor: Chrome/Edge needed for open/save; paste/copy still works | Paste text manually |
| File open | User cancels picker | No error banner; workspace remains unchanged | Continue working |
| File open | Some files rejected | Compact summary listing rejected file names/reasons near toolbar or tab strip | Continue with accepted files |
| File tab | Clean | File status row says `已载入` or `已保存 HH:mm` | Edit or save later |
| File tab | Dirty | Tab badge and file status row say `未保存` | Click `保存` or `保存全部` |
| File save | Saving | Save button disabled or busy, status row says `保存中...` | Wait |
| File save | Permission needed | Status row says `需要重新授权后才能保存` | Click save again to trigger permission |
| File save | Handle missing | Status row says `需要重新打开文件后才能保存` | Re-open file or download copy |
| File save | Disk conflict | Warning banner in input pane with `重新载入文件`, `仍然覆盖`, `下载副本` | Choose one explicit recovery path |
| Save all | Partial success | Compact summary: saved, skipped clean, conflicts, permission needed, failed | User acts on listed tabs |
| Clear all | After click | Workspace resets to one empty tab; local warning cleared if possible | Continue with empty workspace |
| Debug | `?debug` absent | No debug UI | None |
| Debug | `?debug` present | Debug panel below results with parser counts, warnings, dropped candidates | Inspect parser behavior |

State placement rules:

- Editor-affecting warnings live in the input pane, close to the editor.
- Copy-specific failures live beside the failed copy control.
- File conflict is the only high-prominence warning banner; it can overwrite disk data.
- No modal dialogs for normal parser/storage/file states. Use inline recovery controls.
- Destructive `清空全部` remains visible, but visually separate from routine tab/file actions.

### Pass 3: User Journey and Emotional Arc

Score: 6/10 -> 9/10 after fixes.

The tool is for developers handling logs, configs, commands, and local files while debugging. The emotional goal is not delight through decoration; it is calm control.

```text
STEP | USER DOES                         | USER SHOULD FEEL        | UI SUPPORT
-----|-----------------------------------|-------------------------|-------------------------------
1    | Opens Text Preview                | "I know where to start" | Empty editor + visible sample/open actions
2    | Pastes messy text                 | "Nothing left my machine" | Local/no-upload chips stay visible
3    | Scans extracted objects           | "I can grab the exact value" | Line-grouped results + adjacent copy buttons
4    | Switches between contexts         | "My work is separated" | Named tabs, counts, dirty/file badges
5    | Opens local files                 | "This is still local" | File-backed tab title + explicit save controls
6    | Hits save conflict                | "I am protected from overwriting" | Inline conflict banner with three clear choices
7    | Copy is blocked                   | "I am not stuck" | Manual copy fallback with exact value
8    | Clears workspace                  | "The sensitive text is gone" | One empty tab and no lingering sentinel storage
```

Time-horizon design:

- First 5 seconds: user should see a normal QHelper tool shell, one empty editor, result area, and the three trust chips. No tour, no hero, no explanatory wall.
- First 5 minutes: user should discover tabs, file-backed tabs, save-all, and copy-all through visible controls and concise labels, not documentation.
- Long-term use: repeated debugging sessions should feel predictable. The same actions stay in the same zones; warnings appear close to the pane they affect; destructive cleanup remains explicit.

Copy voice:

- Use utility language: `本地解析`, `不上传`, `本机保存`, `未保存`, `冲突`, `需授权`, `需重开`, `保存失败`.
- Avoid product-y text such as `智能提取`, `一站式`, `解锁`, `强大能力`, or long onboarding copy.
- Empty states should be short and action-oriented, e.g. `粘贴文本后会在本地提取可复制对象。`

### Pass 4: AI Slop Risk

Score: 7/10 -> 10/10 after fixes.

Classifier: APP UI.

Hard rules for this workbench:

- No hero section inside the tool. The first screen is the usable editor/results workspace.
- No decorative card grid. Results are line-grouped rows and compact grouped sections, not repeated cards.
- No gradient/orb/blob/background illustration treatment.
- No oversized marketing title. The visible workbench title should be compact; the QHelper breadcrumb already provides page identity.
- No generic `智能`, `高效`, `一站式`, `强大`, `现代化` UI copy.
- No icon-in-colored-circle decoration. Icons only clarify actions/status.
- No nested cards. Use panes, dividers, rows, sticky headers, badges, and inline alerts.
- No one-note purple/blue-purple palette. Use QHelper tokens plus restrained status colors.
- No centered-everything layout. This is a scanning and editing surface; align content to the work grid.

Allowed visual vocabulary:

- Thin dividers and pane boundaries.
- Compact badges for trust/file status/result counts.
- Sticky summary row in the result pane.
- Inline alerts only where recovery is needed.
- Monospace value rendering for source text and extracted values.
- Square or low-radius controls matching QHelper's existing button style.

### Pass 6: Responsive and Accessibility

Score: 5/10 -> 9/10 after fixes.

Responsive contract:

| Viewport | Layout | Required behavior |
|---|---|---|
| Desktop >= 1024px | Sidebar shell + horizontal split panes | Input/result panes default near 48/52 or 50/50; both panes scroll internally, not the whole page |
| Tablet 861-1023px | Horizontal split if content remains usable | Toolbar may wrap once; tab strip remains horizontal scroll; pane resize handle remains available |
| Narrow <= 860px | Vertical stack | Input pane above result pane; result pane starts below a stable editor height; no horizontal page scroll |
| Small mobile <= 620px | Vertical stack with compact actions | Toolbar uses two rows maximum; action labels may shorten but accessible names stay complete |

Stable sizing:

- `ToolWorkspaceShell` keeps the page at `h-screen overflow-hidden`; the tool workbench must fit inside `tool-page-main`.
- Editor and result pane each own their scroll area; avoid body-level scroll inside the tool shell.
- CodeMirror line wrapping must be enabled, and long values in results must break/truncate without forcing horizontal scroll.
- Tab strip scrolls horizontally and must not resize the split panes when tab titles are long.

Keyboard and focus order:

```text
QHelper shell controls
  -> workbench toolbar actions
  -> workspace tablist
  -> active tab rename/delete/new/clear controls
  -> editor
  -> result summary copy-all
  -> row copy buttons
  -> file conflict recovery buttons, when present
  -> debug panel controls, only when visible
```

Accessibility requirements:

- Workspace tabs use `tablist` / `tab` semantics with labels such as `项目 A，12 个对象`.
- Icon-only controls must have visible tooltip text and `aria-label`.
- Copy buttons need full accessible names: `复制 192.168.80.45`, `复制全部对象`.
- Copy success/failure and destructive-confirmation hints use a polite live region.
- Conflict state uses an inline alert pattern with clear title and recovery buttons.
- Visible focus ring must use QHelper focus/ring tokens and meet contrast expectations.
- Body text is at least 16px. Extracted value text may be 13-14px monospace because it is data-dense code text, but it must have sufficient contrast.
- Touch targets for primary mobile controls and copy actions should be at least 44px. If visual buttons remain compact on desktop, mobile CSS must increase hit area.
- Do not rely on color alone for file status or parser warnings; pair color with text labels such as `未保存`, `冲突`, `需授权`, `失败`.
